# 中转站真伪检测平台产品文档

## 1. 产品背景

当前 AI 编程相关中转站数量快速增加，常见宣传口径包括：

- 声称是官方 Claude Code 渠道
- 声称是 Kiro、AWS、GPT 官方转发
- 声称是“官转”
- 使用其他模型或逆向能力伪装成目标模型能力

普通用户很难从营销话术中分辨：

- 这个 `baseUrl` 对应的到底是什么协议和能力
- 这个 `apiKey` 对应的是否真是对外宣传的模型家族
- 站点是否稳定、是否标准兼容、是否长期可信
- 同一个群组或站点历史口碑如何

因此需要一个独立的检测平台，用户只需要提交第三方公开的 `baseUrl + apiKey`，系统自动做探测、分析和评分，并把结果以可理解的方式返回给用户，同时沉淀出中转站和分组维度的红黑榜。

## 2. 产品目标

本产品要解决的核心问题不是“证明它绝对官方”，而是：

1. 帮普通用户快速识别明显异常或高风险的中转站。
2. 帮用户判断站点宣称与实际能力是否一致。
3. 通过多次探测结果沉淀出站点和分组的历史信用画像。
4. 用结构化结果代替截图、口口相传和主观评价。

## 3. 目标用户

### 3.1 直接用户

- 使用 Claude Code、GPT、Kiro、AWS 等编程模型的个人用户
- 购买或试用各类中转站服务的普通开发者
- Telegram、QQ群、论坛等社区中的模型讨论用户

### 3.2 间接用户

- 做榜单整理、渠道测评、自媒体评测的人
- 提供聚合检测服务的网站运营者
- 想验证自己渠道是否被误解的站点运营者

## 4. 核心价值

### 4.1 对普通用户

- 降低被“假官转”“套壳模型”“错误宣传”误导的概率
- 用统一检测标准理解不同站点
- 快速查看某站点和某分组的历史表现

### 4.2 对社区

- 建立可复用的检测结果标准
- 形成群组和站点维度的公开声誉体系
- 逐步积累异常样本，提升识别能力

### 4.3 对产品自身

- 以结构化探测数据为核心资产
- 后续可扩展成模型测评、渠道监控、社区举报等更完整的平台

## 5. 产品定位

这是一个“中转站真实性与可信度检测平台”，不是聊天平台，也不是模型网关。

它的主要职责是：

- 接收用户提交的中转站信息
- 发起标准化探测
- 对结果进行解释与打分
- 输出检测报告
- 汇总形成红榜和黑榜

## 6. 核心问题定义

平台需要回答以下几个问题：

1. 这个站点是否具备标准 OpenAI 兼容能力。
2. 这个站点返回的模型列表更像哪个模型家族。
3. 站点宣称的渠道是否与实际检测结果一致。
4. 这个站点是否存在明显异常，例如鉴权失败、响应不是 JSON、没有模型列表、协议结构异常。
5. 同一站点或同一分组在历史上表现如何。

## 7. MVP 产品范围

当前 MVP 的产品范围聚焦于“探测、回显、排行”三件事。

### 7.1 当前交付形态

当前产品不是单一后端服务，而是一个前后端分离的 Web 应用：

- `frontend/`
  面向普通用户的检测台，负责提交探测、查看检测结果、查看站点红黑榜和分组红黑榜
- `backend/`
  面向系统能力的探测服务，负责请求第三方中转站、评分、落库和聚合排行

当前技术形态上，已经具备：

- 浏览器可操作的检测页面
- 独立后端 API 服务
- 可单独部署的前后端工程
- 支持后续继续演进为详情页、搜索页和后台管理能力的结构基础

当前数据层使用 PostgreSQL，用于保存探测记录、排行样本和后续可能扩展的详情页数据。

### 7.2 输入

用户提交：

- `stationName`：站点名称
- `groupName`：分组或社群来源，可选
- `baseUrl`：中转站地址
- `apiKey`：待测试的 key
- `claimedChannel`：站点宣称渠道，可选，例如 `cc`、`gpt`
- `expectedModelFamily`：用户预期模型家族，可选，例如 `claude`

### 7.3 系统行为

系统执行：

- 尝试请求 `GET /v1/models`
- 如果失败，再尝试 `GET /models`
- 解析响应内容与模型 ID
- 推断模型家族
- 判断 OpenAI 兼容性
- 根据规则打出 `trustScore` 和 `verdict`
- 保存探测结果

### 7.4 输出

系统返回：

- 综合结论
- 可信度分数
- 可疑原因
- 支持证据
- 检测到的主模型家族
- 原始响应摘录
- 站点和分组排行

## 8. 核心功能设计

## 8.1 功能一：单次探测

### 用户故事

作为一个普通用户，我拿到某个中转站给我的 `baseUrl + apiKey` 后，希望把它提交给系统，快速知道它到底是不是像宣传的那样。

### 输入要求

- 支持 HTTPS 地址
- 允许用户补充站点名和分组名
- 允许用户填写宣称渠道和期望模型家族

### 输出内容

- `trustScore`
- `verdict`
- `status`
- `primaryFamily`
- `detectedFamilies`
- `modelIds`
- `suspicionReasons`
- `notes`
- `httpStatus`
- `responseTimeMs`

### 产品价值

把“是否可信”拆成可解释的证据，避免只给一个黑箱结论。

## 8.2 功能二：历史记录

### 用户故事

作为一个复查用户，我希望看到最近测试过哪些站点，每次结果如何，以便对比不同样本。

### 核心要求

- 按时间倒序展示
- 支持点击某次探测查看详情
- 后续应支持按站点名、分组名、模型家族筛选

## 8.3 功能三：站点红黑榜

### 用户故事

作为一个还没下单的用户，我希望先看站点整体口碑和检测结果，再决定是否尝试。

### 排行目标

- 红榜：历史平均分高、成功率高、风险少的站点
- 黑榜：平均分低、高风险次数多的站点

### 当前排序维度

- `avg_score`
- `success_rate`
- `high_risk_count`
- `total_probes`
- `last_probe_at`

## 8.4 功能四：分组红黑榜

### 用户故事

作为一个社区观察者，我希望判断某个群组长期分享的中转站质量如何，避免进错群、买错服务。

### 排行意义

- 识别高质量资源来源
- 识别高风险分发渠道
- 为社区讨论提供结构化依据

## 9. 评分体系设计

当前评分是启发式评分，不代表官方认证，只代表“从标准协议和返回结果来看，这个站点的可信程度更高或更低”。

### 9.1 正向信号

- `/models` 接口返回 2xx
- 响应体可被解析为 JSON
- 响应结构符合 OpenAI 兼容格式
- 能提取到模型 ID
- 模型家族和站点宣称一致
- 模型家族和用户预期一致

### 9.2 负向信号

- 401 或 403 鉴权失败
- 请求超时或网络异常
- 返回 HTML 或非 JSON 内容
- 返回结构不是标准模型列表
- 无法识别模型家族
- 宣称 `cc` 但结果更像 `glm` 或其他家族

### 9.3 分层结果

- `trusted`
  分数高，结构标准，探测结果与宣称基本一致
- `needs_review`
  有一定可用性，但仍有疑点，需要人工判断
- `high_risk`
  存在明显异常、结构不标准或宣称冲突，风险较高

## 10. 红黑榜定义

榜单不是“官方认证榜”，而是“基于探测结果和历史统计的相对可信度榜单”。

### 红榜标准

- 平均得分高
- 成功探测比例高
- 高风险记录少
- 样本量足够

### 黑榜标准

- 平均得分低
- 高风险占比高
- 返回结构异常较多
- 宣称与结果冲突较多

### 产品提示

榜单页面必须明确提示：

- 榜单仅基于平台采样结果
- 不保证站点长期稳定
- 不构成消费建议或官方认证

## 11. 用户流程

### 11.1 首次检测流程

1. 用户进入检测页。
2. 填写 `stationName`、`groupName`、`baseUrl`、`apiKey`。
3. 可选填写 `claimedChannel` 和 `expectedModelFamily`。
4. 提交检测。
5. 系统返回评分、结论、证据和可疑点。
6. 系统将本次结果计入站点榜和分组榜。

### 11.2 榜单查看流程

1. 用户进入排行榜页面。
2. 先看站点红黑榜或分组红黑榜。
3. 点击某一项进入详情。
4. 查看该站点或分组的历史探测记录。

## 12. 结果页设计建议

结果页建议分成四个区域：

### 12.1 总结区

- 结论标签
- 分数
- 主模型家族
- 检测时间

### 12.2 风险区

- 可疑原因列表
- 风险等级说明

### 12.3 证据区

- 成功信号列表
- 响应头摘要
- 模型 ID 样本

### 12.4 原始信息区

- 请求的目标地址
- 命中的 endpoint
- HTTP 状态码
- 原始响应节选

## 13. 数据与隐私要求

### 13.1 敏感信息处理

- 不保存明文 `apiKey`
- 只保存 `apiKey` 哈希和掩码
- 原始响应内容只保留截断节选

### 13.2 风险控制

- 不主动调用高消耗聊天接口
- 优先做只读模型列表探测
- 后续应增加速率限制，防止被平台当成扫描器使用

## 14. 非功能需求

### 14.1 可解释性

每个评分结果必须有原因，不能只给结论。

### 14.2 可扩展性

后续要能增加更多模型家族识别规则、更多协议指纹和更多榜单维度。

### 14.3 稳定性

第三方站点响应慢或异常时，平台自身不能整体卡死。

### 14.4 审计性

每次探测结果都应该可以追溯到对应记录。

## 15. 版本规划

### V1：当前 MVP

- 前后端分离结构
- 浏览器检测台
- 单次探测
- 历史记录
- 站点红黑榜
- 分组红黑榜
- Go 后端 API
- PostgreSQL 持久化

### V2：可用版

- 搜索和筛选
- 异步任务队列
- 站点详情页
- 分组详情页
- 分享页和结果快照

### V3：增强版

- 用户匿名上报
- 评论与举报
- 多维度指纹识别
- 长期稳定性监控
- 更完整的模型仿冒检测

## 16. 成功指标

产品初期建议关注以下指标：

- 每日探测次数
- 独立用户数
- 站点覆盖数
- 分组覆盖数
- 被判定为 `high_risk` 的比例
- 用户对检测结果的采纳率
- 用户对榜单的访问率

## 17. 当前产品边界

这个产品当前不解决以下问题：

- 不保证能从技术上绝对证明“官方渠道”
- 不直接判断价格是否合理
- 不直接判断长期售后和退款风险
- 不直接分析真实对话质量
- 不替代用户自己的交易判断

## 18. 一句话总结

这是一个面向 AI 编程中转站的“真实性与可信度检测平台”，通过标准化探测、启发式判分和历史榜单，帮助普通用户更快识别假宣传、套壳模型和高风险渠道。
